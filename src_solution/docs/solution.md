## 1. Архитектура и границы ДВБ

Решение размещено в каталоге `src_solution/` и построено вокруг разделения кода АБУ на доверенную и недоверенную части.

Главная идея решения — уменьшить доверенную вычислительную базу, оставив в ней только код, который принимает решения безопасности, проверяет политики и управляет состоянием миссии. Недоверенная аналитика и вычисления вынесены отдельно и не могут напрямую управлять критическими параметрами буровой установки.

### 1.2. Доверенный домен TCB

Доверенная вычислительная база находится в каталоге:

```text
src_solution/abu/tcb
```

В неё входят только компоненты, отвечающие за безопасность, политики, журналирование и управление миссией.

| Модуль | Назначение |
|---|---|
| `domain.py` | Описание доменов, запросов и ответов между доменами |
| `event_log.py` | Доверенный журнал событий безопасности |
| `mission_control.py` | Доверенный автомат состояний миссии |
| `placeholder.py` | Служебная проверка доступности ДВБ |
| `policies.py` | Политики безопасности и разрешённые взаимодействия |
| `safety.py` | Детерминированные проверки безопасности |
| `security_monitor.py` | Монитор безопасности и точка применения политик |
| `sys/ipc_policies.json` | Описание разрешённых IPC-политик |

### 1.3. Недоверенный домен OTHER

Недоверенный код находится в каталоге:

```text
src_solution/abu/other
```

Он содержит аналитику, эвристики и вычисления с использованием `numpy`.

| Модуль | Назначение |
|---|---|
| `analytics.py` | Фасад недоверенной аналитики |
| `pseudo_ai.py` | Эвристики псевдо-ИИ: риск, режим бурения, вибрации |
| `numpy_workflow.py` | Сглаживание вибраций с использованием `numpy` |

Недоверенный домен не принимает окончательных решений. Его результаты рассматриваются как входные данные и должны проходить через доверенную проверку в `security_monitor.py`.

## 3. Границы доверия и домены

В решении используются следующие домены:

| Домен | Значение | Назначение |
|---|---|---|
| `TCB` | `tcb` | Доверенная вычислительная база |
| `OTHER` | `other` | Недоверенная аналитика |
| `DIGITAL_MINE` | `digital_mine` | Внешний цифровой рудник |
| `REGULATOR` | `regulator` | Регулятор / сертификационная сторона |

Описание доменов реализовано в файле:

```text
src_solution/abu/tcb/domain.py
```

Взаимодействие между доменами описывается объектами:

- `DomainRequest`
- `DomainResponse`

Это позволяет явно фиксировать источник, цель, действие и полезную нагрузку запроса.

## 4. Политики и цели безопасности

### 4.1. Цель безопасности SG-1: запрет прямого управления из недоверенного домена

Недоверенный код из `abu/other` не должен напрямую менять критическое состояние миссии.

Реализация:

```text
src_solution/abu/tcb/security_monitor.py
src_solution/abu/tcb/domain.py
src_solution/abu/tcb/policies.py
```

Все запросы проходят через `SecurityMonitor.mediate()`.

---

### 4.2. Цель безопасности SG-2: ограничение глубины бурения

Система не должна принимать миссию или шаг миссии, если глубина превышает доверенный предел.

Реализация:

```text
src_solution/abu/tcb/safety.py
src_solution/abu/tcb/policies.py
src_solution/abu/tcb/security_monitor.py
```

Проверяемые функции:

```text
enforce_depth_cap()
SecurityMonitor._validate_mission_payload()
```

---

### 4.3. Цель безопасности SG-3: ограничение RPM

Система не должна принимать опасные обороты бурового инструмента.

Реализация:

```text
src_solution/abu/tcb/safety.py
src_solution/abu/tcb/policies.py
src_solution/abu/tcb/security_monitor.py
```

Проверяемые функции:

```text
enforce_rpm_cap()
SecurityMonitor._validate_mission_payload()
```

---

### 4.4. Цель безопасности SG-4: аварийная остановка

При высоком риске или аномальной вибрации доверенная часть должна переводить миссию в аварийное состояние.

Реализация:

```text
src_solution/abu/tcb/safety.py
src_solution/abu/tcb/mission_control.py
```

Проверяемые функции:

```text
should_emergency_stop()
MissionController.tick()
```

---

### 4.5. Цель безопасности SG-5: журналирование событий безопасности

Все значимые события и нарушения политик должны фиксироваться в доверенном журнале.

Реализация:

```text
src_solution/abu/tcb/event_log.py
```

Журнал поддерживает:

- запись событий;
- уровни `INFO`, `WARNING`, `ERROR`, `CRITICAL`;
- кольцевой буфер;
- сохранение полного журнала на диск;
- очистку многострочных сообщений от переносов строк.

## 5. Монитор безопасности

Главная точка применения политик находится в файле:

```text
src_solution/abu/tcb/security_monitor.py
```

Класс:

```text
SecurityMonitor
```

Основной метод:

```text
mediate()
```

Он проверяет:

1. Разрешена ли пара доменов `source -> target`.
2. Разрешено ли действие `action`.
3. Корректна ли полезная нагрузка для миссии.
4. Не превышены ли ограничения по глубине.
5. Не превышены ли ограничения по RPM.
6. Нужно ли записать нарушение в журнал событий.

### 5.1. Разрешённые действия

Разрешённые действия задаются в `SafetyPolicy`:

```text
health
start_mission
mission_tick
ai_suggest
status
```

### 5.2. Разрешённые связи доменов

Разрешённые связи задаются в `allowed_edges`:

```text
OTHER -> TCB
DIGITAL_MINE -> TCB
REGULATOR -> TCB
TCB -> OTHER
```

Если связь отсутствует в политике, запрос отклоняется с причиной:

```text
domain_violation
```

Если действие неизвестно, запрос отклоняется с причиной:

```text
action_not_allowed
```

## 6. Управление миссией

Управление миссией реализовано в файле:

```text
src_solution/abu/tcb/mission_control.py
```

Основные сущности:

| Сущность | Назначение |
|---|---|
| `MissionState` | Доверенное состояние миссии |
| `MissionController` | Контроллер запуска и выполнения миссии |

### 6.1. Запуск миссии

Метод:

```text
start_mission()
```

Перед созданием миссии отправляет запрос через доверенный монитор:

```text
trusted_request(Domain.DIGITAL_MINE, "start_mission", ...)
```

Если монитор отклоняет запрос, миссия не запускается.

### 6.2. Шаг миссии

Метод:

```text
tick()
```

На каждом шаге:

1. Рассчитывается следующая глубина.
2. RPM ограничивается политикой.
3. Формируется запрос `mission_tick`.
4. Запрос проходит через монитор безопасности.
5. Обновляется состояние миссии.
6. Проверяется аварийная остановка.
7. Событие записывается в журнал.

Возможные состояния миссии:

```text
running
completed
blocked
emergency
```

- наличие security-тестов;
- проверка доменных границ;
- проверка политик глубины и RPM.

## 17. Известные ограничения

В текущей версии остаются ограничения:

1. Процессная изоляция доменов не реализована полностью.
2. Критерий C19 получает 2 балла из 3.
3. Есть замечания flake8 / PEP8, поэтому C09 получает 0 баллов.
4. Критерии C20–C22 являются экспертными и автоматически не начисляются.
5. Размер `mission_control.py` превышает 100 LOC, поэтому C23 получает 2 балла из 3.

Решение реализует безопасную архитектуру АБУ с разделением доверенной и недоверенной частей.

- политики безопасности;
